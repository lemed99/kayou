## High Priority Issues (Fix Before v1.0)

### Issue 1: Missing `aria-busy` for Loading State

**Severity:** High
**Category:** Accessibility
**Line(s):** 65-86

**Current Code:**
```typescript
<button
  type={type()}
  class={twMerge(...)}
  disabled={disabled()}
  {...buttonProps}
>
```

**Problem:**
When `isLoading` is true, the button shows a spinner but doesn't communicate the loading state to screen readers. Users relying on assistive technology won't know the action is in progress.

**Solution:**
```typescript
<button
  type={type()}
  class={twMerge(...)}
  disabled={disabled()}
  aria-busy={local.isLoading}
  {...buttonProps}
>
```

**Estimated Effort:** 5 minutes

### Issue 3: Inconsistent Boolean Prop Naming

**Severity:** High
**Category:** API Design
**Line:** 18-22

**Current Code:**
```typescript
export interface ButtonProps extends JSX.ButtonHTMLAttributes<HTMLButtonElement> {
  color?: ButtonColor;
  size?: ButtonSize;
  isLoading?: boolean;  // Uses 'is' prefix
}
// But uses native `disabled` without 'is' prefix
```

**Problem:**
The component uses `isLoading` (with `is` prefix) but relies on native `disabled` (without prefix). This creates inconsistency. Per library conventions, boolean props should use `is/has` prefix.

**Solution:**
Add `isDisabled` as an alias or primary prop:
```typescript
export interface ButtonProps extends JSX.ButtonHTMLAttributes<HTMLButtonElement> {
  color?: ButtonColor;
  size?: ButtonSize;
  isLoading?: boolean;
  isDisabled?: boolean;
}
```

Then in the component:
```typescript
const disabled = createMemo(() => local.isDisabled || local.disabled || local.isLoading);
```

**Estimated Effort:** 10 minutes
